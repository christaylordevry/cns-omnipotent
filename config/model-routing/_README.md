## Model routing config (non-secret, repo-stored)

This folder contains **versioned, non-secret** configuration used by the CNS routing decision engine and surface adapters (Cursor, Claude Code, Vault IO host integration).

- JSON Schemas for validation
- Config files that validate against those schemas
- Reason code registry
- Model alias registry
- Vitest tests that validate shipped config, enforce secret-safety rules, and test the decision engine

## What is allowed to live here (repo config)

- **Defaults** per surface and task category (e.g. `coding|writing|analysis`)
- **Allow/deny lists** (e.g. allowed model aliases per surface; optional tool allow/deny)
- **Reason codes** (an enum-like registry)
- **Model alias registry** (stable aliases → non-secret resolution metadata)
- **Policy version identifiers** (semantic version or policy hash string)

Everything in this folder must be safe to publish to a public fork.

## What must never be stored here

Per CNS security posture, the following are **forbidden** in repo-stored routing config:
- Credentials of any kind (API keys, tokens, auth headers)
- Prompts, responses, tool payload bodies, or any request/response content
- Vault note content (full text), attachments, or extracted snippets
- "Break-glass" / operator override secrets

If something would normally be stored in fields like `apiKey`, `token`, `Authorization`, it does **not** belong here.

## Never log / never echo (routing audit summary only)

Routing-related audit records must remain **content-free**. Allowed fields are limited to a minimal decision record:
- `surface`: `cursor|claude-code|vault-io|unknown`
- `scope`: `session|task|tool`
- `policy_version`: string
- `selected_model_alias`: string (alias only)
- `reason_code`: from `reason-codes.json`
- `fallback_chain`: array of aliases (max 16)
- `operator_override`: boolean

No content fields.

## Allow/deny precedence (determinism rule)

If a routing candidate matches both an **allow** and a **deny** entry, **deny wins**.

This is intentional: the routing engine must be deterministic and fail-closed when policies conflict. The only exception is the operator override flag, which bypasses deny rules while still requiring the alias to exist in the registry.

## Adding a new model alias

1. Edit `model-alias-registry.json`: add a new key under `aliases` with `provider`, `model_id`, and optionally `label`, `capabilities`, and `notes`.
2. The alias name must follow the naming constraints below.
3. If the alias should be usable on specific surfaces, add it to the `allow.model_aliases` array in `policy.defaults.json` for each relevant surface. If no allow list exists, all non-denied aliases are eligible.
4. Bump `registry_version` in `model-alias-registry.json`.

## Adding a new policy rule

1. Edit `policy.defaults.json`.
2. Each surface has a `defaults` object keyed by task category (`coding`, `writing`, `analysis`). Each entry requires `model_alias` (the primary) and `fallback_chain` (ordered list of fallbacks).
3. Optionally add or modify `allow` / `deny` blocks at the surface level. `allow.model_aliases` acts as a positive filter — only listed aliases are eligible. `deny.model_aliases` blocks aliases regardless of the allow list. Deny always wins.
4. Bump `policy_version` when changing policy rules.

## Alias naming constraints

- Lowercase kebab-case only: `^[a-z0-9][a-z0-9-]{1,62}[a-z0-9]$`
- Minimum 3 characters, maximum 64 characters
- Must start and end with a lowercase alphanumeric character
- Interior characters may include hyphens
- Both `policy.schema.json` and `model-alias-registry.schema.json` enforce this pattern consistently

## When to bump versions

| Change type | Which version to bump |
|---|---|
| Add/remove/rename an alias in the registry | `registry_version` in `model-alias-registry.json` |
| Change policy rules (defaults, allow/deny lists, scopes) | `policy_version` in `policy.defaults.json` |
| Add a new reason code | Update both `reason-codes.json` and `reason-codes.schema.json` |

`policy_version` and `registry_version` are independent — bump only the one that applies to your change. Both accept semantic version strings (e.g. `1.0.0`) or content hash strings (e.g. `sha256:<...>`).

## Versioning approach

Policy versioning is intentionally flexible:
- Use a **semantic version string** (e.g. `1.0.0`), or
- Use a **content hash string** (e.g. `sha256:<...>`) generated by your release process.

The schema only requires that `policy_version` is a string, so operators can choose the approach that matches their governance workflow.

## Local validation

Run all schema validation, credential safety, and decision engine tests:

```bash
npm run test:vitest
```

Or equivalently:

```bash
npm test
```

This runs:
- **Schema validation**: all shipped config files validate against their respective JSON Schemas
- **Credential safety**: key names and string values are scanned against a blocklist
- **Allow/deny conflict detection**: shipped examples must not have aliases on both lists
- **Decision engine**: routing logic tests (happy path, deny-wins, fallback walk, operator override, exhaustion)
- **Surface adapters**: alias resolution, config writes, atomic write safety, error handling

## Surface adapters

Surface adapters translate a `DecisionRecord` (returned by the routing decision engine) into the config format each tool consumes. Adapters are **translators only** — they contain no routing logic, no deny/allow evaluation, and no fallback chain walks. Current adapters: Cursor, Claude Code, Gemini CLI.

Each adapter:
1. Receives a `DecisionRecord` and an `AliasRegistry`
2. Resolves the selected alias to a concrete `model_id` via the shared `resolveAlias` utility
3. Writes the resolved model string to the surface's config location
4. Returns an `AdapterResult` — structured success or failure, never throws

### Cursor adapter

**Module:** `src/routing/adapters/cursor.ts`
**Config target:** `.cursor/settings.json` (workspace-level)
**Config key:** `"cns.routing.model"`

The Cursor adapter writes the resolved `model_id` to a `cns.routing.model` key in the project-level Cursor settings JSON. Writing to workspace-level settings (not user-level) ensures per-project model control. The adapter merges into the existing settings object to avoid clobbering non-routing Cursor config.

### Claude Code adapter

**Module:** `src/routing/adapters/claude-code.ts`
**Config target:** `.claude/settings.json` (project-level) or `~/.claude/settings.json` (user-level)
**Config key:** `"model"` (top-level)

The Claude Code adapter writes the resolved `model_id` to the top-level `"model"` key in Claude Code's settings JSON. This is the canonical persistent config method — Claude Code reads it on startup. The `"model"` key accepts full model names (e.g., `"claude-opus-4-6"`) and is at the lowest-priority persistent level, so operators can still override per-session via the `ANTHROPIC_MODEL` env var, `--model` CLI flag, or `/model` command.

### Gemini CLI adapter

**Module:** `src/routing/adapters/gemini-cli.ts`
**Config target:** `~/.gemini/settings.json` (user-level)
**Config key:** `model.name` (nested)

The Gemini CLI adapter writes the resolved `model_id` under the nested `model.name` key in `~/.gemini/settings.json`. The adapter deep-merges into the existing `model` object so that sibling keys (e.g., `model.safety`, `model.temperature`) are preserved.

**Why `~/.gemini/settings.json` over alternatives:**

| Method | Problem |
|---|---|
| `GEMINI_MODEL` env var | Programmatic `process.env` mutation is global — affects all child processes, not scoped to Gemini CLI. Env var persistence (`.bashrc`/`.zshrc`) is fragile and surface-dependent. |
| `--model` CLI flag | Session-only — not persistently injectable by CNS. Would require wrapping every Gemini CLI invocation. |
| `~/.gemini/settings.json` | Machine-writable JSON, scoped to Gemini CLI only, loaded automatically on CLI startup, supports atomic write. |

## Operator override rules

### Setting override

Pass `operatorOverride: true` in the `RoutingContext` when calling `resolveRoutingDecision`:

```typescript
const context: RoutingContext = {
  surface: "cursor",
  taskCategory: "coding",
  operatorOverride: true,
};
```

### What override bypasses

- **Deny rules:** An alias on the `deny.model_aliases` list is normally skipped. With `operatorOverride: true`, the deny check is bypassed and the alias is selected.

### What override does NOT bypass

- **Registry existence:** The alias must still exist in `model-alias-registry.json`. Override cannot conjure an alias that is not registered. If the alias is denied AND not registered, the engine returns `NO_MATCH_FAIL_CLOSED`.

### When to use override vs updating the policy file

- **Override:** Emergency model swap, one-off testing, temporary workaround. Override is a session-scoped escape hatch.
- **Policy update:** Permanent changes to model selection. Edit `policy.defaults.json` and bump `policy_version`.

### Audit trail

Override decisions use the `OPERATOR_OVERRIDE` reason code. Override decisions are always logged at "visible" tier in the routing audit log regardless of whether the provider changed, because overrides are governance-significant events.

### Fallback behavior with override

When `operatorOverride: true`, the engine also applies override semantics during the fallback chain walk: a denied fallback candidate is selected if it exists in the registry. This means override affects both primary and fallback selection.

### How to add a new surface adapter

1. Create a new module at `src/routing/adapters/<surface-name>.ts`
2. Import `resolveAlias` from `./resolve-alias.js` for alias → `model_id` resolution
3. Import `AdapterResult`, `AliasRegistry`, and `DecisionRecord` from `../types.js`
4. Export a single function: `apply<Surface>Adapter(decision: DecisionRecord, registry: AliasRegistry, configPath: string): Promise<AdapterResult>`
5. Use the atomic write pattern (write to `.tmp`, then `rename`) to prevent partial config writes
6. Return `{ ok: false, error, surface }` for all expected failure modes — never throw
7. Add tests to `tests/model-routing/surface-adapters.test.ts` (use temp directories, no live tool calls)
8. Add a subsection to this README documenting the config target and key
9. Bump `policy_version` if the new adapter changes default routing behavior
