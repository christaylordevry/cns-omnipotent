## What went well (keep doing)

- **Single path primitive:** Reusing `resolveVaultPath` across every read tool kept boundary behavior consistent and reviewable.
- **Stable MCP error JSON:** `CnsError` + `callToolErrorFromCns` gave a repeatable pattern for Epic 4 write failures (`PROTECTED_PATH`, `SCHEMA_INVALID`, etc.).
- **Explicit search scope:** Requiring `scope` or `CNS_VAULT_DEFAULT_SEARCH_SCOPE` matches NFR-P2 and avoids “silent full vault” behavior on WSL-class environments.
- **Verification gate honesty:** Fixing `verify.sh` so it detects real npm scripts prevented a false-green gate story from hiding missing `typecheck` / `build`.

## What did not go well (systems and seams, not people)

- **Read policy still “boundary only.”** Story 3.2 Dev Notes deferred a read blocklist; that remains correct for Phase 1 until a story explicitly adds read denies. Teams should not assume reads are policy-complete beyond escape prevention.
- **Two search implementations** (ripgrep vs Node) increase test surface and behavioral drift risk; mitigated by tests, but changes need dual coverage when touching search.
- **`ErrorCode` subset vs architecture §9:** Architecture lists codes the full Phase 1 surface will use; Epic 3 only needed a subset. Epic 4 must extend the enum and keep the JSON contract aligned—no ad-hoc string codes in tools.

## Patterns worth carrying into Epic 4

- **Validate before IO:** Read tools established “resolve → check → read”; writes should follow “resolve → WriteGate → (later PAKE/secrets) → atomic write.”
- **Tool-per-file under `src/tools/`:** Kept merges predictable; write tools should follow the same layout and register from `index.ts`.
- **Fixture-friendly design:** Vault-relative paths and explicit errors make Epic 4 unit tests straightforward against `tests/fixtures/minimal-vault/` when integration stories land.

## Key learnings

1. **Operational safety beats convenience for search defaults.** Forcing an explicit scope (or env default) is the right default for agent-heavy workflows.
2. **Gate and verify scripts like production code.** A broken verify script erodes trust faster than a failing test.
3. **Defer policy until the enforcing story** is fine **if** the deferral is written down—Epic 4 is where write policy becomes executable.
