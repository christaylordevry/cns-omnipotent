## What we delivered (evidence-based)

- Story 1.1 `1-1-constitution-core-and-modular-policy`
  - Core constitution mirror in `specs/cns-vault-contract/AGENTS.md` within the 500 line budget.
  - Policy modularization via `specs/cns-vault-contract/modules/` (Vault IO, Security) so the core remains compact.
  - Automated mirror and budget checks via `tests/constitution.test.mjs` and the verify gate.

- Story 1.2 `1-2-ide-shims-load-the-constitution`
  - Vault deploy templates under `specs/cns-vault-contract/shims/` for Claude Code and Cursor to load `AI-Context/AGENTS.md`.
  - Repo-local Cursor rule under `.cursor/rules/` so this implementation repo self-grounds against the specs mirror.
  - Automated shim checks via `tests/shims.test.mjs`.

- Story 1.3 `1-3-grounding-parity-checklist`
  - Canonical operator checklist in `specs/cns-vault-contract/README.md` for WSL grounding parity (Cursor and Claude Code) and time-to-grounded observation rubric.
  - Automated checklist guard via `tests/grounding-checklist.test.mjs`.

## What went well (keep doing)

- **Single-system identity and map**
  - The constitution establishes an immediately usable vault map, formatting standards, and module loading protocol. That removes repeated “blank prompt” orientation overhead.

- **Progressive disclosure worked**
  - Keeping the core constitution small and pushing detail into modules preserved the always-on context while allowing depth when needed.

- **Docs paired with tests**
  - The team treated shims and checklists as first-class deliverables, and protected them with guard tests. This is exactly the posture needed for multi-surface parity.

- **Scope clarity about Phase 1**
  - Epic 1 cleanly separated vault artifacts and surface shims from later Vault IO runtime work, avoiding premature “tooling sprawl.”

## What did not go well (improve the system, not people)

- **Normative location ambiguity requires discipline**
  - There are multiple mirrors (`specs/` and `_bmad-output/planning-artifacts/`). Tests enforce parity for some files, but the overall “edit here first” rule is easy to violate without automation.

- **Parity is documented, but runtime parity is still future work**
  - Epic 1 proves “constitution loads without paste” for the surfaces, but the end-to-end Vault IO journey parity depends on later epics shipping a working MCP tool surface.

- **Checklists can rot without tight anchors**
  - The guard tests look for required anchors and keywords, but they cannot guarantee every step remains accurate as Cursor and Claude Code change. This needs an intentional maintenance loop.

## Patterns observed across stories

- **Grounding is an ecosystem problem**
  - The epic succeeded because it treated “context load” as a combination of file placement, rules, and tooling defaults, not a prompt-writing exercise.

- **Mirrors need enforcement**
  - When the project relies on mirrors, parity tests are not a nice-to-have. They are the only thing that makes “single source of truth” real in practice.

## Key learnings

1. **If it must be true every session, make it a default.** Shims and always-apply rules beat “remember to paste.”
2. **A constitution without modules will eventually fail.** The 500 line budget forces good information architecture.
3. **Docs without guardrails drift.** Lightweight tests are an effective Phase 1 substitute for heavier automation.
