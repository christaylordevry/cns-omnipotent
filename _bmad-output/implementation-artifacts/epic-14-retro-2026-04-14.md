## Systems lessons

- **Pre-architecture readouts pay rent twice:** they de-risk later Phase 3 stories and they **surface reviewable policy** (logging cardinality, “never log” lists, host-boundary language for Vault IO vs future routers) before any implementation commitment exists.
- **AC wording is part of the threat model:** reconciling “repo may remain untouched” with sprint metadata changes taught the team to say **“no product code”** explicitly so operators do not misread scope boundaries.
- **Cross-phase assumptions belong in the Given clause:** calling out Phase **2.0** vault stability as the precondition for the readout keeps Phase 3 planning honest when Phase 2.1 Brain and mobile charters are still landing in parallel tracks.

## What to carry forward

1. For any **planning-only** Phase 2+/3 story, end the primary artifact on the **mandated closing section** (here: epic breakdown) so AC “concludes/ends” language is trivially auditable.  
2. Keep **surface matrices** and **secret placement buckets** in one table set reviewers can scan without reading narrative twice.  
3. When a readout mentions future **per-tool** routing, label **host-boundary / future** explicitly so Phase 1 MCP semantics are not implied to change.

- **Sprint metadata coupling:** Retrospective keys for earlier epics were debated during the same change set as the readout; capture **operator decisions** (`1A`, `2A` style) in the story file so future retros do not re-open settled policy.
