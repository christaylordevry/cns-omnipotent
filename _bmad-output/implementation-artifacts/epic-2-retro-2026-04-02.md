## What we delivered (evidence-based)

- Story 2.1 `2-1-per-directory-manifests-for-the-folder-contract`: Per-directory `_README.md` manifests across the folder contract, plus a guard test (`tests/folder-contract-manifests.test.mjs`) to prevent silent drift.
- Story 2.2 `2-2-inbox-capture-semantics`: Inbox exception is explicitly documented and protected by a focused test (`tests/inbox-capture-semantics.test.mjs`).
- Story 2.3 `2-3-note-type-routing-table`: Routing table and disambiguation rules for `WorkflowNote` are documented consistently across constitution, spec, and implementer module, with a guard test (`tests/note-type-routing-table.test.mjs`).

## What went well (keep doing)

- **Contract-as-code posture**
  - The epic consistently treated documentation as part of the product surface, and backed it with automated guard tests. This reduces “policy drift” risk, and it will pay off later when Vault IO tools start enforcing these rules at runtime.

- **Explicit disambiguation over implicit heuristics**
  - Defining “project context” as explicit and requiring operator or manifest disambiguation prevents the system from making confident, wrong guesses about routing `WorkflowNote`.

- **Consistency across mirrors**
  - Keeping `specs/cns-vault-contract/AGENTS.md` and the planning mirror in sync (and tested) preserved the “single truth in practice” behavior the project needs.

- **Scope discipline**
  - Epic 2 stayed inside Phase 1 boundaries by making this a documentation + guard epic, not prematurely implementing runtime routing or ingestion features.

## What did not go well (improve the system, not people)

- **Brittle string-based policy tests**
  - The guard tests rely on stable substrings. This is pragmatic but fragile, it can either false-fail on editorial tweaks or false-pass on semantically broken wording that retains the substring.

- **Ambiguity around `_README.md` frontmatter schema**
  - There was a real decision point: whether directory contract manifests must follow PAKE standard keys or can use contract-specific keys. This got resolved in-story, but the outcome is a known seam that Vault IO tooling must respect explicitly.

- **Epic-level state vs story-level completion**
  - `sprint-status.yaml` still shows `epic-2` as `in-progress` even though all Epic 2 stories are `done`. This makes it harder to use sprint status as a single, reliable source for “what is finished.”

## Patterns observed across stories (signals worth institutionalizing)

- **Documentation decisions need enforcement hooks**
  - Every time the epic made a policy decision, the strongest outcomes paired the decision with a test assertion. This is a repeatable pattern for future epics that start in docs before code exists.

- **“Outside Inbox” boundary is the pivotal governance line**
  - Multiple stories reinforced that the primary behavioral split is “Inbox is permissive, everything else is governed.” This should remain the conceptual anchor for Vault IO `WriteGate` and for operator mental models.

## Key learnings

1. **If policy matters, it needs an automated guard.** Doc-only rules are not durable in multi-agent environments.
2. **Explicit context beats inferred context for routing.** Especially for `WorkflowNote`, implicit inference is an error amplifier.
3. **Mirror strategy is only real when it is tested.** Byte-for-byte mirror enforcement prevents subtle divergence.
