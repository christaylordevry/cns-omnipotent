# Epic 22 Retrospective (Partial): Perplexity Formal MCP

**Date:** 2026-04-30  
**Epic:** 22 — Perplexity Formal MCP  
**Scope of this retro:** **Partial** — based on Story **22.1** completion only. Epic 22 remains **in-progress** in `sprint-status.yaml`.

## Psychological safety note (no blame)

This is a **systems/process** retrospective. Any gaps reflect workflow or tooling design, not individual performance.

## What we set out to do (Story 22.1 intent recap)

Replace the ad hoc, direct HTTP Perplexity integration with a **formal MCP-backed** integration so:

- Perplexity becomes a Tier 1 tool alongside Firecrawl/Apify (centralized operator config).
- We reuse known-good MCP adapter patterns.
- We preserve graceful degradation (no Perplexity → sweep continues with `perplexity_skipped` semantics).
- Tests stay deterministic (no live network calls in CI).
- No secrets ever land in tracked files.

## What shipped (Story 22.1 deliverables)

- **MCP-backed Perplexity slot** (stdio MCP server, default `npx -y perplexity-mcp`, tool call by protocol name, typically `search`).
- **Direct HTTP fetch removed** from the slot and chain wiring; remaining ad hoc runner removed per review.
- **Availability gating** preserved (credentials + runnable command checks) and **graceful degradation** semantics retained.
- **Hardening from review** landed:
  - safer parsing for `PERPLEXITY_MCP_COMMAND` / `PERPLEXITY_MCP_ARGS` (reduce env-driven execution surprises),
  - timeouts/abort strategy around MCP connect/call/close to prevent sweep hangs,
  - cleaner error surfacing (operator-actionable, non-duplicative prefixes),
  - stronger tests (less brittle string asserts; coverage for `available=false` and missing command path).
- **Operator touch points** updated:
  - documented `~/.cursor/mcp.json` registration shape (env-based, no secrets),
  - constitution tool roster updated and **kept byte-identical** across both constitution copies.
- **Verification evidence recorded:** `PERPLEXITY_API_KEY="" bash scripts/verify.sh` PASS (deterministic path validated).

## What went well

- **Unification around the Tier 1 MCP pattern.** This replaces “one-off HTTP slot” behavior with the same operator-configured mechanism used elsewhere.
- **Graceful degradation was treated as a first-class contract.** The sweep remains robust under missing config or Perplexity downtime.
- **Security posture improved in two directions:**
  - no secrets in repo,
  - reduced command/args footguns (env-driven execution hardening).
- **Review loop was high leverage.** The patch set caught the kinds of problems that only show up once you treat it as an operational service (hangs, duplicated errors, accidental shadow entrypoints).

## What was hard / where we got lucky

- **Tool naming ambiguity is real.** Host-prefixed names (`mcp__perplexity__search`) differ from MCP protocol tool names (`search`). This is easy to regress if future contributors copy host names into SDK calls.
- **Shadow entrypoints can reintroduce policy drift.** Removing `scripts/run-research-agent.ts`’s direct Perplexity runner was necessary to keep the story intent true (“formal MCP, not ad hoc fetch”), but this class of drift can recur.
- **“Configured” is not the same as “available.”** Operator config lives outside the repo (`~/.cursor/mcp.json`) while code availability is computed inside the repo, which can cause confusion if expectations aren’t explicit in errors and docs.

## Key insights (repeatable lessons)

1. **Treat external research tiers as “operational dependencies,” not libraries.** Build for timeouts, clean error surfaces, and deterministic tests by default.
2. **Separate three names explicitly in adapters:** host tool alias, MCP protocol tool name, and user-facing label. Mixing them causes silent breakage.
3. **Make “no ad hoc fetch” enforceable.** Once the architecture moves to formal MCP wiring, regression should be blocked by automated checks—not tribal memory.
