## What went well (keep doing)

- **Single canonical tool-surface constant:** Registration drift is now detectable in CI, not only during manual review.
- **Fixture-first integration strategy:** Real-vault dependencies were avoided while still testing the full handler-policy-filesystem chain.
- **Gate hardening with failure proofs:** The project now proves negative paths for `verify.sh`, instead of assuming them.
- **Spec and operator docs stitched together:** Root and spec-index docs both point to the same verification gate rule.

## What did not go well (systems and seams, not people)

- **Story status lagged implementation:** Story 6.3 had shipped changes while still marked review, which delayed obvious Epic closure.
- **Optional cross-link task created ambiguity:** The verification-gate cross-link existed, but unchecked task state implied missing work.
- **Packaging-closeout signals were split:** Completion evidence lived across story files, README updates, and sprint status without one explicit Epic 6 close marker until this retro.

## Patterns worth carrying forward

- **Close review-state stories quickly once acceptance is met** so sprint tracking reflects reality.
- **Treat docs parity as a tracked acceptance task** when duplicate mirrors are intentional.
- **Keep negative-path gate tests versioned in-repo** for future refactors of CI scripts.

## Key learnings

1. **Shippability is a system property:** tool registration, integration tests, and verification scripts must all agree, or a green path is not trustworthy.
2. **Fixture realism beats ad hoc mocks for policy-heavy MCP tools:** boundary, secret, and audit guarantees are easier to validate end to end.
3. **Documentation links are part of quality control:** contributors need a fast path from spec/operator docs to the one command that defines release readiness.
