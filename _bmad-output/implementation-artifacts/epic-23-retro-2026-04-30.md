## Psychological safety note (no blame)

This is a **systems/process** retrospective. Any gaps reflect workflow, constraints, or tooling design—not individual performance.

## What we set out to do (Story 23.1 intent recap)

Improve Brain retrieval quality so primary results skew toward **higher-trust notes** by default—without:

- breaking the read-only retrieval posture (query reads index + sibling manifest only),
- leaking sensitive paths/content,
- requiring schema-breaking index changes, or
- regressing existing Brain retrieval behaviors and tests.

## What shipped (Epic 23 deliverables)

From the Story 23.1 record:

- **Stronger PAKE-weighted ranking** beyond status/confidence/verification by incorporating `pake_type` preference weights, bounded and deterministic.
- **Freshness-aware down-ranking** when sibling `brain-index-manifest.json` indicates likely-stale paths via `freshness.estimated_stale_sample`.
- **Explainability mode** (`explain?: boolean`) that returns numeric score components (similarity, multipliers, penalties, final score) while keeping output safe.
- **Hardening from review**:
  - validate/sanitize record `path` before returning results (skip + warn if unsafe),
  - make “missing quality entirely” penalty explainable/consistent in components,
  - improve the “zero vector query” warning to cover non-finite embeddings.
- **No index schema bump / no build-index change required** (kept backward compatibility and minimized moving parts).

## What went well

- **Trust boundary discipline held.** Retrieval remained read-only and index-artifact scoped, consistent with the Brain charter’s “no silent bypass” and “no vault-body reads at query time” posture.
- **Operational safety was treated as part of ranking.** Freshness signals were not just warnings; they meaningfully influenced ranking in a bounded, deterministic way.
- **Explain mode reduced “mystery ranking.”** Numeric component output creates a stable basis for debugging and tuning without exposing note content.
- **Test-first reinforcement.** New behaviors were anchored in small fixture indexes and explicit ordering asserts, keeping regressions unlikely.

## What was hard / where we got lucky

- **Ranking changes are easy to make “feel right” but hard to make provably safe.** Bounded multipliers, deterministic tie-breakers, and safe output constraints needed explicit attention to avoid accidental nondeterminism or leakage.
- **Freshness is inherently noisy.** The manifest’s stale sample is a heuristic; the implementation had to be careful to treat it as a penalty signal—not a correctness oracle.
- **Backward compatibility pressure.** Any temptation to “just add fields” to index records was correctly resisted; old artifacts must remain queryable.

## Key insights (repeatable lessons)

1. **Make trust signals first-class in ranking, but keep them bounded and explainable.** If operators can’t explain the ordering, they won’t trust it.
2. **Warnings are not enough when ranking quality matters.** A safe, bounded penalty can be a better default than “warn but do nothing,” as long as it’s explicit and test-covered.
3. **Safety constraints must be enforced at the output boundary.** Sanitizing/validating `path` on the way out prevents subtle leakage when upstream data is malformed.
