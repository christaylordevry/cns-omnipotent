# Epic 9 Retrospective — Foundation hardening (shared PAKE types, vault root sanity, wikilink scale)

**Date:** 2026-04-12  
**Epic:** 9 (stories 9-1, 9-2, 9-3)  
**Status:** done  

## What the epic delivered

Three small, high-leverage changes deferred from Phase 1 closure: a single source of truth for PAKE `pake_type` literals in `src/pake/schemas.ts` (consumed by MCP registration and list/create paths), a fail-fast guard rejecting OS filesystem root as `CNS_VAULT_ROOT`, and architecture documentation for `vault_move` wikilink-repair scale and the convergence escape hatch (Obsidian CLI vs full-vault scan on fallback).

## Systems lessons

- **Single-sourcing enums** across MCP tool schemas and runtime validation prevents silent drift when new note types land; treat schema modules as part of the public contract, not an implementation detail.
- **“Valid path” is not enough** for vault root: boundary checks that pass on `/` or a drive root are worse than a loud startup failure—operators get a clear `IO_ERROR` instead of a vault that is technically “inside” the root but meaningless.
- **Documentation as risk control:** wikilink repair is correct but potentially expensive on CLI fallback; stating O(n) assumptions and the preference for `CNS_OBSIDIAN_CLI` sets expectations for operators and future performance work without changing Phase 1 code paths.

## Continuity from Epic 8

Epic 8 clarified Nexus vs Vault IO governance in prose. Epic 9 reinforces that **enforcement and operator-grade config behavior live in code and tests**, not only in constitution text—aligned with Epic 8’s “for later epics” note.

## What to carry forward

1. When adding PAKE-facing tools or types, extend `schemas.ts` first, then wire consumers—avoid parallel literal lists.  
2. Pair any future change to `loadRuntimeConfig` or path resolution with a focused config test (pattern established in 9-2).  
3. Keep architecture **Scale** sections updated when move/search/list behavior touches full-vault scans.
