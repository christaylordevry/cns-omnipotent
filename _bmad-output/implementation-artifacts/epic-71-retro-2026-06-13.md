## What Went Well

1. **Audit-driven scope was precise.** Epic 71 did not expand into Reddit OAuth, vault-export PATH, or trends dedup ‚Äî keeping the blast radius small and the definition of done testable.

2. **Incremental landing worked.** Each story introduced a verifiable contract (structured push result ‚Üí retry buckets ‚Üí outcome record ‚Üí Discord-only repair) so the test suite grew with runtime truth, not after-the-fact.

3. **Shared modules reduced duplication.** `digest-retry-eligibility.mjs` and `digest-run-outcome.mjs` became the SSOT for bucket classification and day-level sticky flags ‚Äî 71-4 reused 71-3's `readDayOutcomeRecord` instead of log inference.

4. **Story specs caught dead-code traps early.** 71-4 explicitly flagged that reordering day-record reads without changing the watchdog early-exit predicate would make bucket-4 unreachable ‚Äî a subtle bug that unit tests in isolation would miss.

5. **Sprint tracker synced with code.** Epic and all four stories marked `done` in the same commit as implementation, avoiding the doc-drift the audit originally flagged.

## Challenges

1. **Legacy exit-code posture (67-10) vs 71-2 truth.** Push script "always exit 0" was intentional for fire-and-forget cron; 71-2 had to supersede that for mutation failures while preserving exit 0 for `missing-convex-env` skip paths.

2. **Three retry buckets, not one.** 71-1 required distinguishing abandoned `started`, real Convex mutation failures, and missing-env config failures ‚Äî each with different recovery shapes (full pipeline vs push-only from artifact).

3. **Log vs Convex vs outcome divergence.** Classifier trusted log terminal actions; outcome record had to cross-check Convex when actions like `skipped-already-pushed` appeared without a published row.

4. **Operator install step outside repo.** 71-3's 19:00 outcome-check cron requires re-running `install-morning-digest-cron.sh` on the host ‚Äî repo change alone does not update live crontab.

5. **Hermes skill path still unobserved.** Manual Discord morning-digest skill writes no outcome record (deferred in `deferred-work.md`).

6. **`computeOverall` over-specified source errors as `partial`.** Story 71-3's mapping table and tests encoded "Reddit error + convex/discord ok ‚Üí partial." That matched the written AC but not the operational contract: source errors belong in `sources[]` for diagnostics; the 19:00 gate should only fail when push or Discord failed. A real production run surfaced it; the fix was 3 insertions and 7 deletions (`ba0dc5c`).

## What We'd Do Differently

1. **Mandatory regression for `computeOverall`:** Any story touching outcome computation must include a test that asserts `overall: 'success'` when `convex.ok && discord.ok` with at least one source in `error` status ‚Äî the most common real-world state while 67-2 remains backlog.

2. **Separate diagnostic from gate semantics in AC writing:** Per-source `error` vs `empty` classification (71-3 AC #6) is valuable for observability; degrading `overall` on source errors was an AC overreach that contradicted the epic's alert contract ("operator sees signal only when push/Discord failed").

3. **Production soak before calling observability "live":** `verify.sh` green and cron installed are necessary but not sufficient; one full calendar day with a broken optional source (Reddit) should be an explicit epic exit criterion.
