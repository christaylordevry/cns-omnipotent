## 1) Epic review

### What we set out to do

- Produce one short operator runbook that makes Tier 1 MCP services usable in practice, not just “registered”.
- Standardize env wiring, key rotation hygiene, and operator-run live-call smoke across Cursor and Claude Code.
- Remove “false-green” setups caused by surface differences and naming ambiguity.

### What shipped

- **Vault-first operator module**: `Knowledge-Vault-ACTIVE/AI-Context/modules/mcp-operator-runbook.md`
  - Secrets-safe env patterns
  - Operator-run live-call smoke definition + safe evidence template
  - Key rotation checklist
  - Cursor vs Claude Code differences table
  - Perplexity reference case + pitfalls
- **Discoverability** wired in the operational surface:
  - `Knowledge-Vault-ACTIVE/AI-Context/modules/vault-io.md` now links the runbook module
  - `Knowledge-Vault-ACTIVE/03-Resources/CNS-Operator-Guide.md` now points to the vault module
- **Source-of-truth preserved** in the implementation repo mirror:
  - `specs/cns-vault-contract/modules/mcp-operator-runbook.md`

### What went well

- **Vault-first posture** matched real operator reality, the runbook is where operators already look during incidents.
- **Safety improvements** were surfaced and fixed during review, especially around shell history and evidence capture hygiene.
- **Parity emphasis** (Cursor vs Claude Code) reduced the most common “configured but not actually working” failure mode.

### What did not go well, or was riskier than it looked

- **Secret-handling nuance**: “read from env” patterns still expand secrets at runtime, so shell history and log hygiene must be treated as part of the system, not operator folklore.
- **Naming ambiguity** is broader than Perplexity: host alias naming varies by host and server, which requires explicit “verify tool names in-surface” guidance.

### Key learnings

- **Operational docs must be vault-native**. Anything repo-only becomes a scavenger hunt during key rotation and smoke.
- **Copy/paste-friendly is not the same as safe**. Docs must actively counter shell history and screenshot/log spill risks.
- **Evidence capture is a first-class deliverable**. If it is not standardized and safe, “verification” becomes anecdotal or unsafe.
